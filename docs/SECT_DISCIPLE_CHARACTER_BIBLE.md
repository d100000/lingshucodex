## 设计目标

地图中的“别宗弟子”不能只是随机名字和复用境界头像。他们应当像武侠小说中的江湖人物一样,有门派气质、年岁辈分、外貌辨识点、性格缺陷、旧怨旧恩和可被玩家反复遇见的故事线。

本文件给出 5 个宗门各 40 名弟子,合计 200 名角色。每宗保持男女各半,覆盖外门、内门、核心、真传、巡使、护法、长老等层级。每个角色都带有画像关键词,可用于批量生成专属头像。

## 画像资产规格

硬性规则:

```text
所有弟子头像、人物立绘、NPC 画像必须使用 gpt-image-2 生成。
禁止使用本地占位绘制器、程序化几何头像、SVG 占位图冒充正式头像。
若只是为了跑通数据链路,必须明确标注 placeholder,且不得提交为最终资产。
```

推荐路径:

```text
frontend/public/images/portraits/disciples/{sect_id}/{disciple_id}.png
```

推荐尺寸:

```text
512x768 PNG, 半身或七分身, 面部清晰, 圆形裁切安全, 无文字无水印。
```

统一提示语骨架:

```text
国风仙侠卡牌人物立绘, 中国传统水墨与重彩结合, 深色背景, 人物居中, 半身构图, 面部清晰, 衣纹精致, 可作为游戏 NPC 圆形头像。角色来自{宗门}, {性别年龄}, {画像关键词}, 气质为{性格关键词}, 背景融入{宗门意象}, 无文字无水印。
```

执行脚本:

```text
scripts/generate_disciple_image2_portraits.py
```

`scripts/generate_disciple_assets.py` 只允许生成 `disciples.json`,不得生成头像。

| 层级 | 等级段 | 年龄段 | 江湖含义 | 常见剧情功能 |
|---|---:|---|---|---|
| 外门 | Lv 4-25 | 14-24 岁 | 初入山门,还保留凡心 | 引路、误会、求助、轻度切磋 |
| 内门 | Lv 26-60 | 20-45 岁 | 门派中坚,最常游历 | 地图常见 NPC、支线主力 |
| 核心 | Lv 61-85 | 35-80 岁 | 可代表宗门处理外务 | 跨宗门冲突、护送、试炼 |
| 真传 | Lv 86-120 | 60-160 岁 | 掌握门派秘传 | 主线人物、宿敌、盟友 |

年龄不是单纯外貌。高境界修士可驻颜,所以“年龄段”主要影响说话方式、江湖阅历和人际关系。

## 主干故事线

1. 青澜古碑线: 沧澜与青冥共同守护古碑,碑文缺了三句。玄机宗说缺句与林泽叛逃有关,天机阁说那是机关密钥,月隐宫说自己见过原碑。
2. 天衍图线: 天机阁和玄机宗各持半卷推演图。两派年轻弟子互相看不起,但真正的图灵藏在一名月隐女弟子的记忆中。
3. 书火案线: 青冥藏书楼百年前失火,月隐宫被诬为纵火者。真凶可能是青冥内部守旧派,也可能是天外道君投影。
4. 叛徒林泽线: 沧澜叛徒林泽不只是魔化,他曾被派去查五宗黑卷交易。每宗都有一名弟子与他有交情。
5. 黑卷商人线: 神秘商人把各宗秘传做成“黑卷”。玩家遇见的行商型 NPC 有概率牵出这条线。
6. 月下名册线: 月隐宫保存许多被篡改的记忆名册。天机阁负责机关锁,玄机宗负责解读顺序,青冥派负责辨伪。
7. 外门失踪线: 五宗外门弟子陆续失踪,看似小案,实为天外裂简在挑选低阶宿主。
8. 五宗问剑线: 每隔十年五宗年轻弟子共赴问剑台。切磋不伤命,却会改变宗门友好度和后续 NPC 态度。

## 江湖事件生成系统

目标不是手写死 5000 条事件,而是建立“事件语法”。事件由门派性格、人物性格、关系链、地点、触发物、解决方式和结果包组合生成。这样每个玩家的世界可以独立演化,并且每个门派对其他四派都能稳定产生至少 5000 种不同事件。

### 事件规模保证

对任意有方向的宗门关系 `A -> B`,事件组合如下:

```text
25 个事件家族
× 10 个发生场景
× 5 个触发原因
× 4 个解决走向
× 5 个结果包
× 3 个稀有度修饰
= 75,000 种方向性事件变体
```

五宗每宗面对另外四宗:

```text
4 个目标宗门 × 75,000 = 300,000 种方向性事件变体 / 宗门
```

系统上线时不必把 75,000 条全部存成文本。只需存模板和随机种子,按 `event_variant_id` 即时生成。若 UI 或策划要求“每宗门固定可见 5000 种事件”,则从 75,000 中按权重抽样缓存 5000 个即可。

同门事件也可生成:

```text
25 个事件家族
× 12 个同门触发原因
× 5 个门内位置
× 4 个解决走向
× 5 个结果包
× 3 个稀有度修饰
= 90,000 种同门事件变体 / 宗门
```

## 下一轮机制

当前主城只外露一个时间推进按钮:

```text
下一轮
```

不要在主流程里并列显示 `下一回合`、`下一天`、`下一轮`、`打坐` 等入口。它们会制造选择噪音。MVP 统一为右下角“下一轮”。

| 按钮 | 时间推进 | 后台计算量 | 前端展示 |
|---|---:|---:|---|
| 下一轮 | 1 天 | 50 名角色, 500 个事件 | 展示个人轻量恢复、宗门关系变化和重点人物变化 |

用户要求的目标流程:

```text
玩家点击“下一轮”
-> 前端结算器从五宗 200 名弟子中按权重抽取 50 名角色
-> 以这 50 名为核心,生成并结算 500 个事件
-> 更新人物好感度、修为、等级、年龄、状态、故事线进度
-> 主角修为不变,只恢复部分气血/灵气并减少疲劳
-> 写入本地事件日志和待同步队列
-> 生成本轮最重要的 20 条江湖简报
```

### 抽取 50 名角色的权重

不是纯随机,而是“玩家能感觉到江湖在动”的加权随机:

```text
基础权重 = 1
+ 与玩家同宗 +2
+ 与玩家最近互动过 +4
+ 身上有未完成故事线 +3
+ 关系好感绝对值 >= 70 +2
+ 处于受伤/失踪/追杀/黑卷状态 +5
+ 角色等级接近玩家 ±10 级 +2
```

### 500 个事件的分配

```text
200 个: 被抽中 50 人之间的直接事件
150 个: 被抽中角色与其关系链人物的事件
80 个: 宗门级事件,影响多个角色
50 个: 主干故事线事件
20 个: 玩家相关传闻或可接取任务
```

### 结算顺序

1. 年龄推进: 所有 200 名弟子按时间粒度增加 age_days。
2. 选择活跃角色: 抽 50 名。
3. 生成候选事件: 每个活跃角色生成 10-20 个候选。
4. 过滤前置条件: 例如失踪者不能参加问剑,重伤者更容易触发疗伤。
5. 结算 500 个事件: 轻事件可批量聚合,重事件单独写日志。
6. 应用结果包: 好感、修为、等级、状态、故事线。
7. 生成简报: 只把高重要度事件返回前端。

### 等级、年龄、修为增长

角色独立拥有自己的成长曲线。成长不应太快,否则 200 人会迅速失控。

```text
age_days += round_time_days
cultivation += event.cultivation_delta + sect_training_bonus
if cultivation >= level_threshold(level):
    level += 1
    cultivation -= threshold
```

建议阈值:

```text
level_threshold(level) = 80 + level * level * 6
```

年龄显示:

```text
visible_age = base_age_years + floor(age_days / 360)
```

高境界驻颜:

```text
apparent_age = min(visible_age, 18 + floor(level / 4))
```

死亡和退场建议先不默认开启。早期可用 `missing / injured / secluded / corrupted` 替代死亡,避免玩家世界线过早断裂。

## 下一轮的体感设计

“下一轮”不应只是一次刷新按钮。它应当是一段短促的江湖推演回放,让玩家感觉五宗弟子在自己看不见的地方继续生活、修炼、结仇、结缘。

### 体验目标

玩家每次点击后,应明确感受到四件事:

```text
世界正在自己运转。
熟悉的人真的经历了事情。
门派之间的关系正在变化。
这一轮留下了可追踪的后果。
```

### 江湖推演回放流程

建议单次回放控制在 5-15 秒。时间太短没有仪式感,太长会阻碍玩家连续推进。

```text
1. 地图轻微暗下,时间从当前日推进到目标日。
2. 五个门派区域依次亮起,根据事件密度产生强弱不同的波动。
3. 被抽中的活跃角色头像快速闪过,表示这些人卷入本轮江湖。
4. 若出现突破、重伤、结仇、结缘、失踪、背叛等事件,弹出命运卡。
5. 回放结束后进入“本轮江湖简报”。
```

地图表现不需要展示 500 个事件。关键是让玩家看到“哪里在动”“谁在动”“动完以后有什么后果”。

### 三层信息展示

第一层是江湖头条,只展示 3-5 条。头条必须是最重要的变化,优先包括门派关系、核心人物、玩家熟人、故事线推进。

```text
月隐谷与玄寂宗关系恶化: 黑卷线索外泄,双方宗门好感 -18。
沧澜剑宗沈照溪救下天机阁程折雪,二人关系变为“生死之交”。
青冥山许问秋闭关破境,升至 Lv 42,但因此被旧敌盯上。
```

第二层是门派变化。每个门派一张简洁面板,用于呈现本轮整体趋势。

```text
沧澜剑宗
声望 +3
与天机阁关系 +8
与月隐谷关系 -4
本轮突破弟子: 3 人
本轮受伤弟子: 2 人
新增恩怨: 5 条
```

第三层是人物关系变化。用头像、箭头和关系标签展示“谁影响了谁”。

```text
周虚空 -> 陆照眠
关系: 旧识 -> 争执旧识
好感: 42 -> 27
原因: 二人在青冥山外因黑卷线索交锋。
```

### 事件展示筛选

前端可以真实结算 500 个事件,但不能把 500 条全部展示给玩家。建议展示结构:

```text
3-5 条江湖头条
10-20 条重要事件
30 条普通传闻
其余事件只结算属性,不直接展示
```

每个事件结算后得到一个 `importance_score`:

```text
importance_score =
  角色稀有度
+ 是否与玩家认识
+ 是否涉及高好感或低好感关系
+ 是否造成等级突破
+ 是否改变门派关系
+ 是否推进故事线
+ 是否产生新关系标签
+ 是否出现受伤、失踪、背叛、结缘等命运状态
```

UI 只消费摘要对象,而不是完整事件堆:

```json
{
  "world_headlines": [],
  "sect_changes": [],
  "character_highlights": [],
  "relationship_changes": [],
  "rumors": [],
  "hidden_events_count": 437
}
```

### 江湖传闻流

简报页应保留一组可滚动的传闻流。传闻文本不必全是大事,可以用不同类型让世界更有生活感。

```text
【门派风波】玄寂宗疑似截获月隐谷密信,双方暗线弟子开始互相试探。
【人物际遇】程折雪夜观星盘,推演出青冥山三日后将有异象。
【关系变化】沈照溪与陆照眠因一场误会好感下降 9 点。
【修为突破】许问秋闭关七日,修为大涨,等级提升至 Lv 42。
【旧事回响】白石渡旧案再次被提起,当年在场的三名弟子关系出现波动。
```

传闻类型建议:

| 类型 | 作用 |
|---|---|
| 门派风波 | 告诉玩家宗门大势正在变化 |
| 人物际遇 | 强化角色正在独立生活 |
| 关系变化 | 让关系链变得可见 |
| 修为突破 | 反馈成长和竞争压力 |
| 旧事回响 | 让历史事件再次影响当前世界 |
| 暗线传闻 | 埋伏后续任务或隐藏剧情 |

### 后果可追踪

每条高重要度事件都应回写到相关角色的简报里。玩家之后打开人物详情时,能看到这件事留下的痕迹。

例如本轮出现:

```text
沈照溪在白石渡救下程折雪。
```

沈照溪的人物简报应新增:

```text
程折雪: 生死之交,好感 +76
最近事件: 曾在白石渡救下程折雪。
```

程折雪的人物简报也要反向显示:

```text
沈照溪: 救命恩人,好感 +82
最近事件: 白石渡遇袭时被沈照溪救下。
```

这样事件就不是一次性文案,而是进入玩家世界的长期记忆。

### 增强体感的界面部件

| 部件 | 设计 |
|---|---|
| 地图热区 | 本轮事件越多的门派区域越亮,若发生冲突则出现红色震动 |
| 宗门连线 | 门派关系大幅变化时,连线从金、白、灰、红之间变化 |
| 命运卡 | 突破、受伤、失踪、结仇、结缘、背叛时单独弹出 |
| 人物闪回 | 重要角色头像闪过,点击可进入人物简报 |
| 江湖榜单 | 展示本轮风云人物、最大恩怨、最快突破、最危险角色 |
| 旧事回响 | 过去事件在后续传闻中被再次提及 |

### 离线优先实现建议

“下一轮”的全部推演应放在前端 Web Worker 中完成,避免阻塞界面。

```text
主线程:
  负责按钮、地图动画、简报展示、人物详情刷新。

Web Worker:
  负责抽取角色、生成事件、结算结果、计算重要度、生成摘要。

IndexedDB:
  保存完整玩家世界、关系变化、人物状态、最近事件和待同步队列。

云端:
  只保存压缩后的世界快照、重要事件摘要和同步版本号。
```

点击后数据流:

```text
玩家点击“下一轮”
-> 主线程向 Web Worker 发送当前 world_id 和推进粒度
-> Web Worker 从 IndexedDB 读取当前世界状态
-> Web Worker 结算 500 个事件
-> Web Worker 写回人物、关系、门派、故事线变化
-> 主线程读取摘要对象并播放江湖推演回放
-> 若在线,把压缩后的 patch 放入云端同步
```

### 设计风险

| 风险 | 解决方式 |
|---|---|
| 事件太多导致玩家疲劳 | 只展示头条、重点、传闻,其余只结算 |
| 前端结算卡顿 | 使用 Web Worker,并把 500 个事件分批结算 |
| 离线数据被篡改 | 单机体验可接受;若未来有排行和交易,关键数据交给后端校验 |
| 存档越来越大 | 普通事件只保留最近 100-300 条,永久保存重要事件和状态结果 |
| 玩家看不懂变化来源 | 每条关系变化都写明原因,并能回到人物简报追踪 |

最终原则: 玩家不需要看到所有 500 个事件,但必须相信那 500 个事件真实发生过,并且已经改变了这个世界。

## 修行驱动玩法设计

当前执行口径: 主城不再展示配置型“下一轮”面板,也不再提供“本轮目标 / 修行地点 / 风险姿态 / 一键调息”等前置配置。主流程只保留一个右下角大按钮:

```text
下一轮
```

这个按钮的定位不是“修为收益按钮”,而是“世界时间推进按钮”。玩家点击后,主角状态只做轻量调养:

```text
修为: 不变
气血: 按当前缺口恢复一部分
灵气: 按当前缺口恢复一部分
疲劳: 减少一部分
```

同时,五宗后台故事照常推进: 每轮仍按玩家专属世界种子抽取约 50 名角色,结算约 500 条事件,更新弟子关系、宗门传闻、角色修为、伤势和暗线。

核心循环改为:

```text
查看状态 -> 点击下一轮 -> 江湖快报 -> 状态小恢复 -> 新传闻/新钩子
```

### 打坐入口的最终处理

旧方案中的“打坐”与“一键调息”会和“下一轮”抢入口,因此从主城删除。

```text
删除:
- 打坐独立主按钮
- 一键调息
- 下一轮配置面板
- 本轮目标 / 修行地点 / 风险姿态选择器
```

保留:

```text
- 主城右下角“下一轮”
- 点击后展示江湖快报
- 点击后恢复少量气血与灵气,减少疲劳
- 不增加修为,不消耗 Key
```

后续如果要重新引入打坐,只能作为战斗外的专题玩法或墨炉章节题材,不能再次成为主城并列按钮。

### 下一轮按钮的体感要求

按钮不显示复杂预测,避免把玩家拉回表格配置。它只需要让玩家明确三件事:

```text
1. 世界进入下一日或下一轮。
2. 自己获得轻量恢复。
3. 五宗关系和人物故事继续变化。
```

结算反馈必须包含:

```text
- 气血变化
- 灵气变化
- 疲劳变化
- 修为不变
- 3 条以内重点江湖事件
- 五宗事件数量概览
```

### 疲劳度: 决策压力而非硬惩罚

疲劳度建议命名为 `心神`、`劳损` 或 `道心负荷`。它不应简单阻止玩家行动,而应改变风险和收益。

| 疲劳区间 | 状态 | 效果 |
|---:|---|---|
| 0-30 | 清明 | 修炼效率 +10%,突破稳定性提高 |
| 31-60 | 平稳 | 无明显影响 |
| 61-80 | 疲惫 | 突破成功率下降,争执事件概率提高 |
| 81-100 | 过劳 | 可能受伤、走火、心魔增加、误判他人意图 |

疲劳来源:

```text
连续战斗
连续冲关
长途外出
炼器炼丹失败
参与高强度宗门事件
压制伤势继续修行
```

疲劳恢复:

```text
静心调息
睡眠休整
同门照料
药物治疗
前往灵气稳定地点
完成心结事件
```

设计原则: 疲劳不是让玩家“不能玩”,而是让玩家需要安排节奏。玩家想继续冒险可以继续,但世界会用更高风险回应他。

### 生命值: 从血条升级为伤势系统

生命值不宜只承担“归零死亡”的功能。早期版本建议用伤势状态替代死亡,让它生成剧情。

| 状态 | 影响 | 可能事件 |
|---|---|---|
| 轻伤 | 战斗效率下降 | 同门探望、仇敌试探 |
| 内伤 | 修炼效率下降 | 医修介入、旧伤复发 |
| 经脉受损 | 突破风险上升 | 寻药、修复经脉、因祸得悟 |
| 中毒 | 每轮损失生命或灵气 | 找解药、追查毒源 |
| 心魔 | 随机影响选择和关系判断 | 梦魇、误会、执念、顿悟 |
| 重伤闭关 | 无法参与部分事件 | 被保护、被算计、闭关破后而立 |

下一轮时,伤势可以触发:

```text
自然恢复
同门照料
医修治疗
仇敌趁虚而入
旧伤复发
因伤悟道
欠下人情
暴露弱点
```

伤势系统的目标是把失败变成故事来源,而不是简单惩罚。

### 灵气: 从资源条变成环境选择

灵气建议分为两层:

```text
自身灵气: 当前角色可立即调用的资源。
地脉灵气: 当前地点提供的环境资源。
```

不同地点应适合不同玩法:

| 地点 | 加成 | 风险或副作用 |
|---|---|---|
| 沧澜剑峰 | 剑修突破、战斗感悟 | 更易触发切磋和争胜 |
| 天机观星台 | 推演、悟道、预知线索 | 心神消耗更高 |
| 玄机机关室 | 炼器、阵法、机关事件 | 失败可能受伤或损坏器物 |
| 青冥药谷 | 疗伤、炼丹、解毒 | 易卷入采药和救人事件 |
| 月隐寒潭 | 潜行、幻术、心法 | 更易触发秘密和误会 |
| 中立市集 | 交易、传闻、人情 | 容易遇到骗局和旧敌 |
| 野外秘境 | 高收益修炼和奇遇 | 受伤、失踪、遭遇战概率上升 |

玩家点击前选择地点,点击后系统根据地点结算个人成长和江湖事件。这样“去哪里修行”本身就成为策略。

### 推动玩家点击下一轮

按钮只显示“下一轮”,不要再把预计修为、地点选择或风险姿态堆在主城首屏。它的吸引力来自“世界要继续发生事情”,而不是“我点了就涨修为”。

示例:

```text
下一轮
第 12 日
```

按钮点击后显示江湖快报:

```text
气血 +36
灵气 +120
疲劳 -18
修为 不变

南宫照衣替玄无声隐瞒了一段旧债。
月照衡替温照海修好一件旧物。
李寒月与夜阑宫主在白石渡切磋三招。
```

这些反馈会让玩家自然想点下一轮,因为下一轮承诺的是“江湖继续流动”。

### 每轮必须有个人结果和世界结果

每次点击后,至少给玩家两类明确反馈:

```text
个人调养结果:
气血恢复、灵气恢复、疲劳下降、修为不变。

江湖变化结果:
玄寂宗与月隐谷关系恶化,周虚空失踪,青冥山许问秋突破。
```

玩家需要同时感到:

```text
我恢复了一点,可以继续行动。
世界也在变化。
```

### 下一轮数值建议

基础结算只做恢复,不做修为收益:

```text
气血恢复 = min(气血缺口, 气血上限 * 0.16)
灵气恢复 = min(灵气缺口, 灵气上限 * 0.18)
疲劳下降 = min(当前疲劳, 疲劳上限 * 0.22)
修为变化 = 0
```

五宗事件仍按 50 名角色、500 个事件结算;弟子个人修为和等级可以在后台增长,但主角修为只由战斗、本命书、墨炉成章等系统提供。

### 让下一轮形成连续钩子

每轮结算结束后,系统应主动留下 1-3 个新的钩子:

```text
关系钩子: 某人因你本轮选择对你产生好感或误会。
成长钩子: 你距离突破只差一小段修为。
伤势钩子: 你的内伤需要特定药材。
门派钩子: 某宗冲突将在三日后升级。
秘密钩子: 你听见一个涉及旧案的名字。
```

这些钩子会变成下一次点击前的动机。

### 最终策划原则

不要用疲劳度限制玩家点击,而要用“时间推进才能结算收益”吸引玩家点击。

理想状态是玩家心里想:

```text
我再点一轮,看看能不能突破。
我再点一轮,看看那封信到了没有。
我再点一轮,看看沈照溪和程折雪后来怎么样了。
我再点一轮,看看五宗会不会开战。
```

当玩家产生这种期待时,“下一轮”就不是按钮,而是观察整个江湖继续运转的入口。

## 与爽感玩法优化总案的融合建议

`GAMEPLAY_OPTIMIZATION_MASTER_PLAN.md` 的重点是“操作爽、战斗爽、成长爽、Key 使用有价值感”。本圣经的重点是“五宗弟子、关系链、江湖推演、玩家专属世界”。两者不应互相替代,而应形成双循环:

```text
短循环: 探索 -> 战斗 -> 掉落 -> 强化 -> 推荐下一步
长循环: 本轮安排 -> 下一轮 -> 江湖回放 -> 关系变化 -> 新钩子
```

短循环负责让玩家每次点击都有即时反馈;长循环负责让玩家相信世界在持续变化。

### 融合定位

| 系统 | 总案中的定位 | 融入五宗圣经后的作用 |
|---|---|---|
| Key 接入信任层 | 降低 BYOK 焦虑 | 区分“本地江湖推演”和“燃灵成章”消耗,避免玩家以为每轮 500 事件都会烧 Key |
| 新手三步入道 | 3 分钟完成首个闭环 | 新手在第一战后立刻接触“下一轮”,看到自己成长和五宗变化 |
| 小怪极速战 | 高频材料和爽感来源 | 作为短循环入口,战后结果可转化为江湖传闻或个人修行素材 |
| 敌人意图 | 让出招有判断 | 可反哺人物性格,例如谨慎者更爱防守,激进者更爱重击 |
| 连招与斩杀 | 制造操作高光 | 高光可写入本命书,也可成为 NPC 对玩家评价的依据 |
| 战后高光 | 告诉玩家获得了什么 | 直接生成“下一轮”的待结算事项和推荐目标 |
| 装备与技能构筑 | 让掉落有去处 | 门派性格决定推荐流派,法宝也可成为关系事件的争夺物 |
| 破境仪式 | 境界高光 | 破境不仅改变玩家数值,也应引发同门评价、他宗关注和江湖传闻 |
| 推荐目标 | 解决不知道做什么 | 是短循环和长循环之间最重要的桥 |
| 秘境 Roguelite | 5-10 分钟完整一局 | 秘境结果可变成五宗事件源,例如失踪、夺宝、救援、线索外泄 |
| 山海经与 Boss 线索 | 长期收集目标 | Boss 线索可挂接到宗门暗线、弟子旧案和门派冲突 |
| 本命书与战报 | 沉淀个人故事 | 记录玩家高光,同时引用江湖推演中的重要 NPC 和门派变化 |

### 离线体验与 Key 消耗的边界

本游戏若采用离线优先方案,必须明确两类行为:

```text
本地结算:
下一轮、五宗事件、关系变化、普通传闻、角色成长。

燃灵成章:
本命书章节、Boss 正传、破境章、秘境总结章、玩家主动要求的高质量叙事。
```

前者应尽量在前端完成,不依赖网络,也不消耗 Key。后者才调用模型,并在调用前给玩家明确预估。

这样可以同时满足:

```text
离线可玩。
服务器压力低。
玩家不担心每次点下一轮都会消耗大量 token。
AI 仍然用在最有记忆点的时刻。
```

### 新手融合流程

建议把“新手三步入道”扩展为“五步入江湖”:

```text
1. 验证 Key,完成灵脉契约。
2. 拜入门派,生成玩家专属五宗世界。
3. 完成第一场极速小怪战。
4. 使用掉落完成一次强化或技能精进。
5. 点击第一次“下一轮”,看到个人成长和江湖变化。
```

第五步非常关键。玩家不能只知道“我能打怪”,还要在第一天就感受到“我所在的世界也在运转”。

### 战斗如何进入江湖关系链

每场战斗结束后,不只是给修为和材料,还应生成可被长循环读取的结果:

```json
{
  "battle_highlight": {
    "enemy_id": "fox_001",
    "max_damage": 312,
    "combo": "凝神破绽",
    "execute_used": true,
    "rare_drop": "狐尾护符残片"
  },
  "world_hooks": [
    {
      "type": "npc_notice",
      "target": "canglan_012",
      "reason": "听闻玩家一剑斩杀山林狐妖"
    },
    {
      "type": "boss_clue",
      "target": "boss_momo",
      "clue": "黑墨符"
    }
  ]
}
```

下一轮时,这些 `world_hooks` 可触发:

```text
同门称赞或嫉妒。
他宗弟子邀战或试探。
Boss 线索推进。
山海经研究提升。
本命书可手动成章。
```

### 推荐目标成为中枢

推荐目标不应只服务战斗,还应同时服务修行和江湖:

```text
战斗推荐: 再击败 2 只狐妖可炼制狐尾护符。
修行推荐: 你的疲劳偏高,建议下一轮静心调息。
关系推荐: 程折雪对你有新传闻,下一轮交际可能触发邀约。
门派推荐: 月隐宫与玄寂宗冲突升温,可前往中立市集探听。
突破推荐: 修为已满,可选择冲关破境。
```

主界面只显示 1 个主推荐和少量次入口,避免信息过载。

### 策划分析意见

融合时最重要的不是继续增加系统,而是让每个系统都能回答三个问题:

```text
它是否让玩家下一次点击更有期待?
它是否让玩家战后知道自己变强在哪里?
它是否把个人行为写回江湖世界?
```

如果一个功能只增加菜单,却不能推动战斗、成长、关系或下一轮,应暂缓。

优先融合顺序建议:

| 阶段 | 目标 | 说明 |
|---|---|---|
| 第一阶段 | 推荐目标 + 战后高光 + 下一轮入口 | 先把短循环和长循环接起来 |
| 第二阶段 | 极速小怪 + 意图 + 连招/斩杀 | 提升高频点击爽感 |
| 第三阶段 | 打坐目标、疲劳、伤势、灵气地点 | 强化下一轮的策略深度 |
| 第四阶段 | 装备、技能流派、破境仪式 | 让成长有长期构筑 |
| 第五阶段 | 秘境、山海经、Boss 线索、本命书强化 | 扩展中期留存 |

最大风险是系统同时上线太多,导致主城变成按钮堆。应始终坚持:

```text
一个主目标。
一个下一步按钮。
一条清晰收益。
一个正在变化的江湖。
```

## 整体策划 Review 与 AI 执行细则

本节用于约束后续 AI 实现、拆任务和写代码时的判断。当前设计已经具备强世界观和强差异化,但最大的风险不是内容不足,而是入口过多、系统并行、玩家不知道下一步该做什么。

最终产品目标应收束为一句话:

```text
玩家点击主城右下角“下一轮”,看到自己轻量恢复、熟人变化、江湖继续运转。
```

### 核心 Review 结论

| 级别 | 问题 | 判断 | 执行要求 |
|---|---|---|---|
| P0 | 主循环入口分散 | `下一回合 / 下一天 / 下一轮 / 打坐 / 探索` 并列会让玩家迷路 | MVP 只保留一个主城右下角大按钮: `下一轮` |
| P0 | 打坐与下一轮抢功能 | 无冷却连击打坐会把玩家训练成狂点恢复,削弱世界推进 | 删除打坐与一键调息入口,由“下一轮”提供轻量恢复 |
| P0 | Key 消耗边界不清 | 玩家会担心 500 个事件都消耗 token | 明确: 本地推演免费,燃灵成章才消耗 Key |
| P1 | 500 事件结算边界不统一 | 前文写前端 Worker,后文若写后台事务会误导实现 | 统一为前端 Worker 结算,后端只同步 patch 和重要摘要 |
| P1 | 200 名弟子初期记忆负担高 | 世界很大,但玩家无法立刻关心所有人 | 新手期只推 5-8 人的熟人圈 |
| P1 | 爽感来源冲突 | 即时狂点爽与回合期待爽同时存在会互相稀释 | 以回合期待爽为主,即时反馈只做辅助 |
| P2 | 主城信息过载 | 再加五宗世界后容易变成按钮墙和信息墙 | 首屏只保留状态、核心入口和右下角“下一轮” |

### MVP 玩家主路径

任何新功能如果不能服务这条路径,都应暂缓。

```text
1. 验证 Key,完成灵脉契约。
2. 拜入门派,生成玩家专属五宗世界。
3. 完成第一场极速小怪战。
4. 使用掉落完成一次强化、炼制或技能精进。
5. 第一次点击“下一轮”。
6. 看到气血/灵气/疲劳小恢复、熟人变化、宗门传闻。
7. 获得新的江湖钩子。
```

这条路径的设计目标:

```text
3 分钟内知道能玩什么。
10 分钟内完成一次变强。
30 分钟内认识至少 3 个值得记住的 NPC。
第一天结束前理解“我动一下,江湖也动一下”。
```

### 用户游玩路径与心理变化

| 阶段 | 玩家心理 | 高风险 | 应提供的体验 |
|---|---|---|---|
| 0-3 分钟 | 不信任,怕 Key 花钱 | BYOK 焦虑、说明模糊 | 灵脉契约、成本说明、可删除 Key、可见预算 |
| 3-10 分钟 | 想确认好不好玩 | 系统太多,不知道点哪 | 第一战、第一掉落、第一次强化 |
| 10-30 分钟 | 想知道下一步 | 目标发散 | 主推荐目标 + 下一轮入口 |
| 30-60 分钟 | 开始认人 | 200 人太多 | 熟人圈、重点关系、人物命运卡 |
| 第一天后 | 想看世界变化 | 事件疲劳 | 江湖头条、旧事回响、可追踪后果 |
| 留存期 | 想形成个人故事 | 只有刷怪会腻 | 本命书、破境章、Boss 线索、关系链长期变化 |

AI 执行时应优先服务玩家当前心理。不要在新手期展示完整系统,也不要在玩家尚未建立熟人圈前大量抛陌生名字。

### 爽感与成瘾性设计

本项目不应依赖“无冷却狂点”制造低级成瘾。更适合的爽感是“可预期但不完全确定”的修行期待。

推荐爽感层级:

| 层级 | 时间 | 爽点 | 示例 |
|---|---:|---|---|
| 即时反馈 | 0-300ms | 数字、动画、状态条变化 | 修为 +120、疲劳 -8、好感 +6 |
| 短期待 | 5-15s | 江湖回放、命运卡、突破提示 | 某弟子突破、某宗冲突升级 |
| 中期待 | 1-5 分钟 | 推荐目标、材料闭环、熟人邀约 | 再打一只可炼法宝、下一轮信件送达 |
| 长期待 | 1 天以上 | 关系线、Boss 线索、本命书 | 旧案浮出、破境成章、五宗大战 |

应强化的心理钩子:

```text
临近突破: 我再点一轮可能升级。
等待回信: 那封信下一轮会不会到?
关系悬念: 沈照溪和程折雪后来怎么样了?
门派异动: 月隐宫和玄寂宗会不会开战?
旧事回响: 上次救下的人会不会记得我?
```

应避免的成瘾方式:

```text
无冷却重复点击。
只靠飘数字刺激。
不给决策只给收菜。
用疲劳硬卡玩家。
用长文本强迫等待。
```

### 熟人圈机制

200 名弟子是内容资产,但玩家初期只能记住少数人。系统应为每个玩家生成一个初始熟人圈。

初始熟人圈建议包含 5-8 人:

| 类型 | 数量 | 作用 |
|---|---:|---|
| 同门引路人 | 1 | 教玩家理解本宗 |
| 同辈竞争者 | 1 | 提供成长压力 |
| 温和友人 | 1 | 提供正反馈和安全感 |
| 他宗熟人 | 1-2 | 打开跨宗关系 |
| 神秘线索人 | 1 | 连接旧案或 Boss 线索 |
| 潜在暧昧或宿敌 | 1 | 制造情感牵引 |
| 受玩家行为影响者 | 1 | 让玩家感到行动有后果 |

早期江湖推演的可见事件应优先围绕熟人圈展开。其他 190 多名弟子可以在后台活动,但不应同时挤进玩家视野。

### 功能合理性判定标准

AI 新增任何功能前,必须用以下四问自检:

```text
它是否让玩家下一次点击更有期待?
它是否让玩家知道自己变强在哪里?
它是否把个人行为写回江湖世界?
它是否减少而不是增加玩家的选择迷路?
```

若答案少于 3 个“是”,该功能应暂缓。

### 主界面信息架构

主城首屏不应展示所有系统。推荐首屏只保留:

```text
1. 当前角色状态: 生命、灵气、疲劳、修为进度。
2. 当前主城核心入口: 探索、背包、技能、本命书等。
3. 右下角大按钮: 下一轮。
4. 点击后江湖快报: 少量恢复 + 五宗事件摘要。
5. 最近一条江湖大事。
```

其他内容进入二级页面:

```text
背包
技能
本命书
山海经
Boss 线索
五宗关系
修行日志
设置与 Key
```

主界面的判断标准:

```text
玩家一眼能知道现在状态。
玩家一眼能知道下一步。
玩家一眼能看到世界有变化。
```

### AI 实现优先级

后续 AI 执行任务时,按以下顺序拆解:

| 顺序 | 任务 | 目标 |
|---:|---|---|
| 1 | 删除打坐和一键调息入口 | 避免与“下一轮”抢主流程 |
| 2 | 实现右下角“下一轮” | 让主城拥有唯一世界推进按钮 |
| 3 | 实现下一轮 MVP | 前端结算事件,展示江湖快报 |
| 4 | 实现熟人圈 | 让玩家先记住少数 NPC |
| 5 | 战后高光写入 world_hooks | 把短循环接入长循环 |
| 6 | IndexedDB 本地世界状态 | 支撑离线和低服务器压力 |
| 7 | 云端同步 patch | 支撑多设备和备份 |
| 8 | 命运卡和江湖回放 | 强化体感 |
| 9 | 关系简报回写 | 让事件可追踪 |
| 10 | 本命书燃灵成章 | 把重要事件升格为文学记忆 |

### 必须避免的实现误区

```text
不要先做完整 500 事件,先做 30-50 个可见事件的 MVP。
不要让下一轮每次都请求 LLM。
不要把所有 NPC 同时推给玩家。
不要把疲劳做成不能玩的硬门槛。
不要把打坐做成独立高频刷数值按钮。
不要让主城出现超过一个主 CTA。
不要把普通事件全部永久写云端。
不要让 Key 文案和实际存储行为不一致。
```

### 验收指标

| 指标 | 合格线 |
|---|---:|
| 新用户完成 Key 验证后到第一战 | 3 分钟内 |
| 第一战后看到明确下一步 | 100% |
| 第一次下一轮展示江湖变化 | 100% |
| 首日玩家认识的重点 NPC | 至少 3 人 |
| 主界面主 CTA 数量 | 1 个 |
| 下一轮默认 Key 消耗 | 0 |
| 普通事件云端完整保存比例 | 低于 10% |
| 玩家能追溯重要关系变化原因 | 100% |

## 玩家专属世界与数据库方案

每个玩家创建角色时,系统应初始化一个专属江湖世界。不要把 200 名弟子完整文本复制给每个玩家;静态资料放全局模板,玩家只保存可变状态。

### 全局静态表

```sql
CREATE TABLE disciple_templates (
  id TEXT PRIMARY KEY,
  sect_id TEXT NOT NULL,
  name TEXT NOT NULL,
  gender TEXT NOT NULL,
  base_rank TEXT NOT NULL,
  base_level INTEGER NOT NULL,
  base_age_years INTEGER NOT NULL,
  portrait_id TEXT NOT NULL,
  appearance TEXT NOT NULL,
  personality TEXT NOT NULL,
  story_hook TEXT NOT NULL,
  trait_json TEXT NOT NULL DEFAULT '{}'
);

CREATE TABLE disciple_relation_templates (
  source_id TEXT NOT NULL,
  target_id TEXT NOT NULL,
  relation_type TEXT NOT NULL,
  base_affinity INTEGER NOT NULL DEFAULT 0,
  note TEXT NOT NULL DEFAULT '',
  PRIMARY KEY (source_id, target_id, relation_type)
);

CREATE TABLE event_templates (
  id TEXT PRIMARY KEY,
  family TEXT NOT NULL,
  scope TEXT NOT NULL,
  base_weight REAL NOT NULL DEFAULT 1,
  template_text TEXT NOT NULL,
  precondition_json TEXT NOT NULL DEFAULT '{}',
  effect_json TEXT NOT NULL DEFAULT '{}'
);
```

### 玩家世界表

```sql
CREATE TABLE player_worlds (
  user_id TEXT PRIMARY KEY,
  world_seed TEXT NOT NULL,
  current_round INTEGER NOT NULL DEFAULT 0,
  current_day INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT
);

CREATE TABLE player_disciple_state (
  user_id TEXT NOT NULL,
  disciple_id TEXT NOT NULL,
  level INTEGER NOT NULL,
  cultivation INTEGER NOT NULL DEFAULT 0,
  age_days INTEGER NOT NULL DEFAULT 0,
  status TEXT NOT NULL DEFAULT 'active',
  location TEXT NOT NULL DEFAULT '',
  flags_json TEXT NOT NULL DEFAULT '{}',
  story_json TEXT NOT NULL DEFAULT '{}',
  updated_at TEXT,
  PRIMARY KEY (user_id, disciple_id)
);

CREATE TABLE player_disciple_relation_state (
  user_id TEXT NOT NULL,
  source_id TEXT NOT NULL,
  target_id TEXT NOT NULL,
  relation_type TEXT NOT NULL,
  affinity INTEGER NOT NULL DEFAULT 0,
  flags_json TEXT NOT NULL DEFAULT '{}',
  last_event_id TEXT,
  updated_at TEXT,
  PRIMARY KEY (user_id, source_id, target_id, relation_type)
);

CREATE TABLE player_sect_state (
  user_id TEXT NOT NULL,
  sect_id TEXT NOT NULL,
  affinity INTEGER NOT NULL DEFAULT 0,
  tension INTEGER NOT NULL DEFAULT 0,
  resources INTEGER NOT NULL DEFAULT 0,
  scandal INTEGER NOT NULL DEFAULT 0,
  flags_json TEXT NOT NULL DEFAULT '{}',
  PRIMARY KEY (user_id, sect_id)
);

CREATE TABLE player_event_log (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  round_no INTEGER NOT NULL,
  day_no INTEGER NOT NULL,
  event_variant_id TEXT NOT NULL,
  importance INTEGER NOT NULL DEFAULT 0,
  visibility TEXT NOT NULL DEFAULT 'rumor',
  actor_ids_json TEXT NOT NULL DEFAULT '[]',
  delta_json TEXT NOT NULL DEFAULT '{}',
  summary TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 索引建议

```sql
CREATE INDEX idx_player_disciple_user_status
  ON player_disciple_state(user_id, status, level);

CREATE INDEX idx_player_relation_user_source
  ON player_disciple_relation_state(user_id, source_id, affinity);

CREATE INDEX idx_player_relation_user_target
  ON player_disciple_relation_state(user_id, target_id, affinity);

CREATE INDEX idx_player_event_user_round
  ON player_event_log(user_id, round_no, importance);
```

### 初始化策略

玩家创角后:

1. 创建 `player_worlds` 行,生成 `world_seed`。
2. 为 200 名弟子写入 `player_disciple_state`,只保存等级、修为、年龄、状态等可变字段。
3. 从 `disciple_relation_templates` 复制基础关系到 `player_disciple_relation_state`。
4. 初始化五宗 `player_sect_state`。
5. 预生成第 0 日 20 条传闻,让世界一开始就有动静。

### 大量玩家支撑方式

如果用户很多,可以进一步压缩:

| 数据 | 存储方式 |
|---|---|
| 弟子姓名、外貌、故事 | 全局模板,所有玩家共享 |
| 玩家世界中的等级/年龄/状态 | 每玩家 200 行,可接受 |
| 关系 | 只存与模板不同的 delta;首次查询时用模板 + delta 合并 |
| 事件日志 | 只长期保存高重要度事件,低重要度事件按月归档或聚合 |
| 500 个事件结算 | 前端 Web Worker 结算并写入 IndexedDB;云端只接收压缩 patch 与重要摘要 |

事件日志不必保存 500 条完整长文本。建议:

```text
重要度 >= 70: 保存完整 summary,前端可展示
重要度 30-69: 保存结构化 delta,summary 可延迟生成
重要度 < 30: 只写聚合计数,不写明细
```

## 互动规则建议

地图遇见弟子时,系统不应只随机意图,而应结合角色关系:

| 条件 | 倾向 |
|---|---|
| 玩家所属宗门与 NPC 宗门友好度高 | 寒暄、交易、传道概率提高 |
| 角色有跨宗旧情或债务 | 触发专属对话,可能给线索 |
| 玩家多次击败同宗 NPC | 敌意提高,但好战型角色可能更敬重玩家 |
| 玩家帮助翻案或救人 | 对应宗门角色改变台词和开放故事线 |
| 玩家持有黑卷、残页、名册碎珠 | 相关角色主动靠近或追杀 |
